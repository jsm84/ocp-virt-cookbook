Configuring Static IPs for VMs

Overview

This tutorial covers the methods available for assigning static IP addresses to virtual machines in OpenShift Virtualization. While DHCP is the default and simplest approach, many workloads require predictable, fixed IP addresses for integration with external systems, compliance, or legacy application requirements.

You will learn how to configure static IPs using five distinct methods:

  • Cloud-init networkData — configure networking at first boot (Linux VMs)

  • UDN primary network with persistent IPs — cluster-managed IP assignment via OVN-Kubernetes

  • DHCP reservations — bind a fixed IP to a MAC address on an external DHCP server

  • QEMU guest agent with nmcli — apply static IPs post-boot via the guest agent

  • Sysprep (Windows VMs) — configure networking through unattend.xml

Each method has different trade-offs in terms of automation, flexibility, and operational overhead. Choose the one that best fits your environment.

What You Will Learn

  • The trade-offs of assigning static IPs in OpenShift Virtualization

  • How to configure static IPs at first boot using cloud-init networkData

  • How to use UDN primary networks with persistent IPs for cluster-managed addressing

  • How to use DHCP reservations with fixed MAC addresses

  • How to apply static IPs post-boot using the QEMU guest agent and nmcli

  • How to configure static IPs for Windows VMs using Sysprep

  • How to choose the right method for your environment

Trade-offs and Considerations

Before assigning static IPs, understand the implications:

Loss of cluster-managed networking

When you assign a static IP on a secondary network, the VM may lose access to OpenShift Services and Routes that rely on the default pod network. The primary interface (connected to the pod network) should retain its DHCP-assigned address to preserve cluster connectivity.

No cluster awareness of assigned addresses

Static IPs bypass OVN-Kubernetes IPAM. The cluster does not track these addresses, which can lead to IP conflicts if assignments are not managed externally. Maintain an IP address inventory or use an external IPAM solution.

Maintenance overhead

Static IP assignments must be tracked and updated manually (or via external IPAM) as VMs are created, moved, or removed. DHCP with reservations can reduce this burden.

When static IPs are appropriate

  • Legacy workloads that require fixed addresses

  • Integration with external systems (firewalls, load balancers, monitoring) that reference IPs directly

  • Environments without a DHCP server on the target network

  • Regulatory or compliance requirements for address traceability

When to prefer DHCP or OVN-Kubernetes IPAM

  • Ephemeral or frequently recreated VMs

  • Environments with reliable DHCP infrastructure

  • Workloads that communicate exclusively through Kubernetes Services and DNS

Prerequisites

  • OpenShift 4.18+ with OpenShift Virtualization operator installed

  • CLI tools: oc, virtctl

  • At least one configured network (primary pod network)

  • For secondary network scenarios: a NetworkAttachmentDefinition (NAD) configured (see Localnet VLAN Tutorial)

  • For DHCP reservation scenarios: administrative access to an external DHCP server

  • Basic understanding of networking (IP addressing, subnets, gateways, DNS)

Method 1: Cloud-init networkData

Cloud-init is an industry-standard initialization tool that runs during VM boot. The networkData field in the cloudInitNoCloud volume provides dedicated network configuration using netplan-compatible version 2 format.

This is the most common method for Linux VMs in OpenShift Virtualization.

How It Works

  1. The VM boots and cloud-init reads the networkData from the NoCloud datasource.

  2. Cloud-init applies the network configuration before user-space services start.

  3. The configuration persists across reboots.

Configuration Format

Cloud-init network configuration uses version 2 (netplan) format. The version 2 format supports ethernets, bonds, bridges, and vlans as device types. Each device can include addresses, dhcp4/dhcp6, routes, nameservers (with addresses and search), mtu, and match (by macaddress, name, or driver) options.

The basic format is:

networkData: |
  version: 2
  ethernets:
    enp1s0:
      dhcp4: true
    enp2s0:
      dhcp4: false
      dhcp6: false
      addresses:
        - 192.168.100.50/24
      nameservers:
        addresses:
          - 192.168.100.1

In this example, enp1s0 (the primary pod network interface) uses DHCP, while enp2s0 (the secondary interface) receives a static IP.

Default Gateway Configuration

Do not use the gateway4 or gateway6 fields — they are deprecated since netplan 0.103 and cloud-init 23.1. Use routes instead:

networkData: |
  version: 2
  ethernets:
    enp2s0:
      dhcp4: false
      addresses:
        - 192.168.100.50/24
      routes:
        - to: 192.168.100.0/24
          via: 192.168.100.1
      nameservers:
        addresses:
          - 192.168.100.1
        search:
          - example.com

This example also shows how to add DNS search domains with the search field and specific routes with the routes list. Use to: default for a default gateway route (only on secondary interfaces when no default route exists on the primary).

Precedence Order

  1. networkData configuration (highest priority)

  2. Datasource network configuration

  3. Fallback to DHCP (default)

Interface Naming in KubeVirt

KubeVirt uses predictable network interface naming inside guest VMs. Interfaces are named sequentially based on their PCI slot:

  • enp1s0 — first interface (typically the pod network)

  • enp2s0 — second interface (first secondary network)

  • enp3s0 — third interface (second secondary network)

Do not use legacy names like eth0. Use the predictable names (enpXs0) that match the order of interfaces defined in the VM spec.

Important Formatting Rules

When providing networkData in a cloudInitNoCloud volume, keep these rules in mind:

Do not wrap with a network: key

KubeVirt’s networkData field expects the configuration to start directly with version: 2. Do not add a top-level network: wrapper — that format is used in standalone config files but not in the NoCloud datasource networkData field.

Always include userData

Cloud-init will not parse networkData unless userData is also set. Even a minimal userData header (#cloud-config) is sufficient. All examples in this tutorial include userData.

Prefer inline networkData over Secret references

There is a known issue where networkDataSecretRef may not be applied by cloud-init in some configurations. Use the inline networkData field or networkDataBase64 for reliable behavior.

Where Static IPs Apply

Static IP configuration applies to secondary network interfaces only. The primary interface connects to the pod network via masquerade and receives its IP automatically from the cluster — do not assign a static IP to it. The masquerade bridge operates on an internal subnet (10.0.2.0/24) that is translated by NAT to the pod IP. Overriding this with a static address on a different subnet will break connectivity.

All examples in this tutorial configure static IPs on secondary interfaces (enp2s0 and above) while leaving the primary interface (enp1s0) on DHCP.

Scenario: Primary DHCP + Secondary Static IP

This is the standard approach. The primary interface uses DHCP on the pod network, while a secondary interface on a VLAN or bridge network gets a static IP.

Prerequisites: Ensure you have a secondary network configured (for example, a localnet VLAN 100 NetworkAttachmentDefinition (NAD)). See Localnet VLAN Tutorial for setup instructions.

Create a namespace for this tutorial:

oc create namespace static-ip-demo

Download fedora-multi-network-vm.yaml

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-multi-vm
  namespace: static-ip-demo
spec:
  dataVolumeTemplates:
    - apiVersion: cdi.kubevirt.io/v1beta1
      kind: DataVolume
      metadata:
        name: fedora-multi-vm-disk
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
  runStrategy: RerunOnFailure
  template:
    metadata:
      labels:
        kubevirt.io/domain: fedora-multi-vm
    spec:
      domain:
        cpu:
          cores: 2
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - masquerade: {}
              name: default
            - bridge: {}
              name: vlan-network
          rng: {}
        machine:
          type: pc-q35-rhel9.4.0
        memory:
          guest: 4Gi
      networks:
        - name: default
          pod: {}
        - name: vlan-network
          multus:
            networkName: localnet-vlan-100
      volumes:
        - dataVolume:
            name: fedora-multi-vm-disk
          name: rootdisk
        - cloudInitNoCloud:
            userData: |-
              #cloud-config
              user: fedora
              password: fedora123
              chpasswd: { expire: False }
              ssh_pwauth: True
            networkData: |-
              version: 2
              ethernets:
                enp1s0:
                  dhcp4: true
                enp2s0:
                  dhcp4: false
                  dhcp6: false
                  addresses:
                    - 192.168.100.50/24
                  nameservers:
                    addresses:
                      - 192.168.100.1
          name: cloudinitdisk

Method 2: UDN Primary Network with Persistent IPs

User Defined Networks (UDNs) with Layer 2 topology offer a cluster-managed alternative for stable IP addressing. When a UDN is configured as the primary network for a namespace with ipam.lifecycle: Persistent, OVN-Kubernetes allocates IPs from a defined subnet and persists them across VM restarts, stop/start cycles, and live migrations using IPAMClaim custom resources.

This approach eliminates the need for manual static IP configuration or external DHCP servers. The cluster handles IP allocation and guarantees that each VM retains its assigned address.

How It Works

  1. You create a namespace with the k8s.ovn.org/primary-user-defined-network label.

  2. You define a UserDefinedNetwork resource with Layer 2 topology, a subnet, and ipam.lifecycle: Persistent.

  3. VMs deployed in that namespace automatically receive a persistent IP from the subnet — no cloud-init networkData or DHCP reservation is needed.

When to Use This Method

  • You want cluster-managed IP assignment without manual configuration

  • VMs must retain the same IP across restarts and live migrations

  • You prefer not to maintain external DHCP servers or embed IPs in manifests

  • You need namespace-scoped network isolation with custom subnets

For a complete walkthrough including namespace setup, UDN creation, VM deployment, and connectivity verification, see Configuring Layer 2 Primary Networks with UDNs.

Method 3: DHCP Reservations

DHCP reservations (also called static DHCP or fixed-address DHCP) combine the automation of DHCP with the predictability of static IP assignment. The DHCP server maps a specific MAC address to a fixed IP, so the VM always receives the same address.

How It Works

  1. You assign a fixed MAC address to the VM’s network interface in the VM spec.

  2. On your external DHCP server, you create a reservation that maps that MAC to a specific IP.

  3. The VM boots, sends a DHCP request, and receives its reserved address.

When to Use This Method

  • You have an existing DHCP infrastructure and want centralized IP management

  • You prefer not to embed network configuration in VM manifests

  • You need the VM to be portable (the IP follows the MAC, not the manifest)

Assigning a Static MAC Address

To use DHCP reservations, you must set a fixed MAC address on the VM interface. Without this, KubeVirt assigns a random MAC at each boot, breaking the reservation.

Download fedora-dhcp-reserved-vm.yaml

apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-dhcp-reserved-vm
  namespace: static-ip-demo
spec:
  dataVolumeTemplates:
    - apiVersion: cdi.kubevirt.io/v1beta1
      kind: DataVolume
      metadata:
        name: fedora-dhcp-reserved-vm-disk
      spec:
        sourceRef:
          kind: DataSource
          name: fedora
          namespace: openshift-virtualization-os-images
        storage:
          resources:
            requests:
              storage: 30Gi
  runStrategy: RerunOnFailure
  template:
    metadata:
      labels:
        kubevirt.io/domain: fedora-dhcp-reserved-vm
    spec:
      domain:
        cpu:
          cores: 2
        devices:
          disks:
            - disk:
                bus: virtio
              name: rootdisk
            - disk:
                bus: virtio
              name: cloudinitdisk
          interfaces:
            - masquerade: {}
              name: default
            - bridge: {}
              name: vlan-network
              macAddress: "02:00:00:01:00:01"
          rng: {}
        machine:
          type: pc-q35-rhel9.4.0
        memory:
          guest: 4Gi
      networks:
        - name: default
          pod: {}
        - name: vlan-network
          multus:
            networkName: localnet-vlan-100
      volumes:
        - dataVolume:
            name: fedora-dhcp-reserved-vm-disk
          name: rootdisk
        - cloudInitNoCloud:
            userData: |-
              #cloud-config
              user: fedora
              password: fedora123
              chpasswd: { expire: False }
              ssh_pwauth: True
          name: cloudinitdisk

The key difference from the cloud-init method is the macAddress field on the secondary interface. No networkData is needed because the DHCP server handles IP assignment.

Configuring the DHCP Server

On your external DHCP server, create a reservation for the MAC address. The exact configuration depends on your DHCP software.

Example for ISC DHCP (dhcpd.conf):

host fedora-dhcp-reserved-vm {
  hardware ethernet 02:00:00:01:00:01;
  fixed-address 192.168.100.60;
}

Example for dnsmasq:

dhcp-host=02:00:00:01:00:01,192.168.100.60,fedora-dhcp-reserved-vm

After configuring the reservation, deploy the VM:

oc apply -f fedora-dhcp-reserved-vm.yaml

Verification

Access the VM and confirm the IP matches the reservation:

virtctl console fedora-dhcp-reserved-vm -n static-ip-demo

Inside the VM:

ip addr show enp2s0

With a DHCP reservation in place, the output shows the reserved IP on the fixed MAC:

Expected output
3: enp2s0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1400 qdisc fq_codel state UP group default qlen 1000
    link/ether 02:00:00:01:00:01 brd ff:ff:ff:ff:ff:ff
    inet 192.168.100.60/24 brd 192.168.100.255 scope global dynamic enp2s0
       valid_lft 3600sec preferred_lft 3600sec

Without a DHCP server responding, the interface will only have a link-local IPv6 address. The fixed MAC 02:00:00:01:00:01 confirms the macAddress field is applied correctly.

MAC addresses starting with 02: are in the locally administered range and will not conflict with hardware vendor-assigned MACs.

Live Migration Considerations

When a VM with a static IP is live-migrated to another node, the behavior depends on the network type:

  • Bridge-based secondary networks (localnet, Linux bridge): The VM retains its MAC address and IP during migration. The network fabric must support MAC mobility (for example, the switch must update its MAC address table). Static IPs assigned via cloud-init, DHCP reservations, or nmcli persist across migrations.

  • SR-IOV networks: Live migration with SR-IOV requires additional configuration and may not preserve the same network path. Consult the OpenShift Virtualization live migration documentation for details.

The primary pod network interface receives a new pod IP after migration, but this is transparent to the guest OS when using masquerade mode.

Method 4: QEMU Guest Agent with nmcli

The QEMU guest agent allows the host (KubeVirt) to communicate with a running VM. You can use this channel to configure networking after the VM has booted, using standard tools like nmcli inside the guest.

This method is useful when you need to change network configuration on an already-running VM, or when cloud-init is not available or practical.

How It Works

  1. The VM boots with DHCP or no IP on the target interface.

  2. The QEMU guest agent starts inside the VM.

  3. You connect to the VM console (or SSH) and use nmcli to apply a static IP.

  4. The configuration persists across reboots because NetworkManager saves it.

Prerequisites

  • The VM image must include the qemu-guest-agent package (most Red Hat and Fedora images include it by default)

  • The guest agent must be running inside the VM

Applying a Static IP

First, deploy a VM with DHCP on the secondary interface (or no networkData for that interface).

Connect to the VM (for example, the DHCP-reserved VM deployed in Method 2):

virtctl console fedora-dhcp-reserved-vm -n static-ip-demo

Verify the guest agent is running:

systemctl status qemu-guest-agent

If the guest agent is not running, start and enable it:

sudo systemctl enable --now qemu-guest-agent

List the current network connections:

nmcli connection show

Identify the connection name for the target interface (for example, enp2s0 or Wired connection 2).

Apply the static IP configuration:

sudo nmcli connection modify "Wired connection 2" \
  ipv4.method manual \
  ipv4.addresses 192.168.100.70/24 \
  ipv4.dns 192.168.100.1

Activate the updated configuration:

sudo nmcli connection up "Wired connection 2"

Verify the result inside the VM:

ip addr show enp2s0

You can also verify from outside the VM using the VirtualMachineInstance status:

oc get vmi fedora-dhcp-reserved-vm -n static-ip-demo -o jsonpath='{.status.interfaces[*].ipAddress}'

The configuration is saved by NetworkManager and persists across VM reboots.

When to Use This Method

  • Post-deployment network changes on running VMs

  • VMs migrated from other platforms where cloud-init was not used

  • Temporary or dynamic network reconfiguration

  • Troubleshooting scenarios where you need to override cloud-init settings

If the VM was originally configured with cloud-init networkData, cloud-init may overwrite nmcli changes on reboot (depending on the cloud-init configuration). To prevent this, either remove the networkData from the VM spec or configure cloud-init to not manage the target interface.

Method 5: Sysprep for Windows VMs

For Windows virtual machines, Microsoft Sysprep with an unattend.xml answer file provides static IP configuration during the Windows Out-of-Box Experience (OOBE) or specialize pass.

How It Works

  1. You create an unattend.xml file with network settings.

  2. The file is injected into the VM via a sysprep volume.

  3. Windows reads the answer file during setup and applies the network configuration.

Example unattend.xml Network Section

The following snippet configures a static IP on a secondary network interface identified as Ethernet during the specialize pass. In a multi-NIC Windows VM, verify the interface Identifier matches the adapter connected to your target network — use Device Manager or Get-NetAdapter in PowerShell to confirm the name:

<?xml version="1.0" encoding="utf-8"?>
<unattend xmlns="urn:schemas-microsoft-com:unattend">
  <settings pass="specialize">
    <component name="Microsoft-Windows-TCPIP"
               processorArchitecture="amd64"
               publicKeyToken="31bf3856ad364e35"
               language="neutral"
               versionScope="nonSxS"
               xmlns:wcm="http://schemas.microsoft.com/WMIConfig/2002/State">
      <Interfaces>
        <Interface wcm:action="add">
          <Identifier>Ethernet</Identifier>
          <Ipv4Settings>
            <DhcpEnabled>false</DhcpEnabled>
          </Ipv4Settings>
          <UnicastIpAddresses>
            <IpAddress wcm:action="add" wcm:keyValue="1">192.168.100.80/24</IpAddress>
          </UnicastIpAddresses>
          <Routes>
            <Route wcm:action="add">
              <Identifier>0</Identifier>
              <NextHopAddress>192.168.100.1</NextHopAddress>
              <Prefix>0.0.0.0/0</Prefix>
            </Route>
          </Routes>
        </Interface>
      </Interfaces>
    </component>
    <component name="Microsoft-Windows-DNS-Client"
               processorArchitecture="amd64"
               publicKeyToken="31bf3856ad364e35"
               language="neutral"
               versionScope="nonSxS"
               xmlns:wcm="http://schemas.microsoft.com/WMIConfig/2002/State">
      <Interfaces>
        <Interface wcm:action="add">
          <Identifier>Ethernet</Identifier>
          <DNSServerSearchOrder>
            <IpAddress wcm:action="add" wcm:keyValue="1">8.8.8.8</IpAddress>
          </DNSServerSearchOrder>
        </Interface>
      </Interfaces>
    </component>
  </settings>
</unattend>

Injecting the Answer File

Store the unattend.xml in a ConfigMap and reference it in the VM spec as a sysprep volume. For a complete walkthrough of Windows VM deployment with Sysprep, see the OpenShift documentation on creating Windows VMs with Sysprep.

Create a ConfigMap with the answer file:

oc create configmap windows-sysprep --from-file=unattend.xml -n static-ip-demo

Reference it in the VM spec:

volumes:
  - name: sysprep
    sysprep:
      configMap:
        name: windows-sysprep

And add the corresponding disk:

devices:
  disks:
    - cdrom:
        bus: sata
      name: sysprep

When to Use This Method

  • Windows Server VMs that require static IPs

  • Automated Windows deployments with answer files

  • Environments migrating Windows workloads from VMware or Hyper-V

Sysprep runs only during initial Windows setup. For post-deployment IP changes on Windows VMs, use PowerShell remoting or RDP to configure networking manually.

Common Issues and Solutions

networkData wrapped with network: key

Problem: Cloud-init silently ignores networkData because it was wrapped in a network: key.

Solution: In KubeVirt’s cloudInitNoCloud volume, the networkData field must start directly with version: 2. Do not add a top-level network: wrapper. The correct format is version: 2\nethernets:\n …​, not network:\n version: 2\n ethernets:\n …​.

Interface names do not match

Problem: Used eth0 instead of enp1s0 in networkData.

Solution: Use predictable interface names as they appear inside the guest OS. Modern Linux distributions (RHEL 8+, Fedora, Ubuntu 16.04+) use enp1s0, enp2s0, and so on. The order matches the interface order in the VM spec.

No connectivity on secondary interface

Problem: Forgot to disable DHCP on the static interface.

Solution: Always set dhcp4: false and dhcp6: false when using static IPs in cloud-init networkData.

Multiple default routes cause connectivity issues

Problem: Default route set on both primary and secondary interfaces.

Solution: Only set a default route on the primary interface (usually enp1s0). Secondary interfaces should not have a default route.

Cloud-init configuration not applied

Problem: Syntax error in YAML or cloud-init configuration.

Solution: Check cloud-init logs inside the VM at /var/log/cloud-init.log. Verify YAML indentation and use cloud-init status to check for errors.

DNS not resolving on secondary network

Problem: Nameservers not configured or incorrect on the secondary interface.

Solution: Add appropriate nameservers in the networkData for the interface. If the secondary interface is on an isolated network, external DNS may not be needed.

Deprecated gateway4 field generates warnings or fails

Problem: Using gateway4: 192.168.100.1 in networkData generates deprecation warnings or fails on newer cloud-init versions (23.1+).

Solution: Replace gateway4 with a routes entry. Use routes: [{to: default, via: 192.168.100.1}] instead. The gateway4 and gateway6 fields were deprecated in netplan 0.103.

DHCP reservation not working

Problem: VM receives a different IP each boot despite DHCP reservation.

Solution: Verify the macAddress field is set in the VM interface spec. Without a fixed MAC, KubeVirt assigns a random one at each boot. Also confirm the DHCP server reservation matches the MAC address exactly.
